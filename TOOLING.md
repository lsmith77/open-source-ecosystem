# Tooling

This document supports [PROPOSAL.md](PROPOSAL.md), the central document. The schema improvements and companion standards are defined there; this file only catalogs the tooling needed to make them real and is read against them.

Tooling is the substrate of adoption. A schema change that ships without a validator, editor, and parser update is a liability, not a feature — broken or missing tooling is one of the largest adoption barriers and undermines a spec's credibility (see [ROADMAP.md](ROADMAP.md): "Update validators, editors, and client libraries in tandem"). This document maps the tooling ecosystem around `publiccode.yml`: what already exists, what adjacent tools this proposal leans on, and what new or adapted tools the proposal needs.

**Enabling catalogs is the point — catalog adoption is the success metric.** This proposal exists to give catalogs and other consumers (procurement tools, sovereignty checks, assessment registries) richer, comparable, machine-readable data to act on. So it has achieved its goal precisely when catalogs adopt and surface the new fields — not when the spec ships or a file is authored. Catalogs therefore carry equal weight here, not treated as an afterthought to authoring: the producing side (editors, validators, crawlers) exists to feed them, and a field no catalog consumes delivers no value. That is why catalog search/display and cross-registry aggregation are tracked as gaps with the same weight as authoring tools.

**Scope:** tooling across the full lifecycle of the metadata in scope of this proposal — authoring, validating, enriching, crawling, distributing, discovering registries, collecting usage/credit declarations, and **consuming** the data in catalogs and other consumers. General-purpose assessment tools that _produce_ artifacts (security scanners, SBOM generators, accessibility auditors) are _not_ cataloged here — they appear only as reference targets that publiccode metadata points to. But the _consumers_ of that metadata (catalogs, procurement and sovereignty assessors) are squarely in scope: seeing them adopt it is how the proposal knows it has met its goal.

It is a working inventory, not a spec. Anything proposed here must still be checked against the [Design Principles](PROPOSAL.md#design-principles) — especially the rule that maintainers should only be asked to author data that genuinely requires their judgment.

### Data sources, crawlers & signal layers

Tools that **collect, derive, or distribute** the data in scope — not the tools that _produce_ the underlying artifacts. An SBOM generator, a security scanner, or an accessibility auditor is out of scope here; what matters is the tooling that gets such outputs _referenced, collected, and distributed_ through `publiccode.yml` and the registries around it. The supply-chain artifacts the proposal references (SBOMs via [CycloneDX](https://cyclonedx.org/)/[SPDX](https://spdx.dev/), [OpenSSF Scorecard](https://scorecard.dev/) results, [REUSE](https://reuse.software/) status) are **reference targets**, not entries in this catalog.

| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Metadata source / auto-derivation | [ecosyste.ms](https://packages.ecosyste.ms/)                                                                                                   | Largest open dataset of OSS packages, repos, and dependencies; can auto-populate or enrich publiccode metadata and crosswalk to package registries — a feeder for crawler-assisted authoring. |
| Classification vocabulary         | [OSS Taxonomy](https://github.com/ecosyste-ms/oss-taxonomy)                                                                           | The versioned facet vocabulary the proposal's `classification` consumes.                                                                                                                      |
| Crawl / redistribution            | [publiccode-crawler](https://github.com/italia/publiccode-crawler), [OpenCatalogi](https://github.com/OpenCatalogi), EU OSS Catalogue | Crawl publiccode files and redistribute them through catalogs and APIs.                                                                                                                       |
| Signal layer / usage registry     | [openCode.de badges](https://badges.opencode.de/)                                                                                  | Confirmed-adopter and compliance signals; a de-facto usage registry the proposal would formalize via the Registry Discovery Standard.                                                         |

Four leverage points stand out, because much of the work is _extension_, not greenfield:

- The **editor, validator, and crawler already exist** ([publiccode-editor](https://github.com/italia/publiccode-editor), [publiccode-parser-go](https://github.com/italia/publiccode-parser-go), [publiccode-crawler](https://github.com/italia/publiccode-crawler)) and need extending, not building.
- The **Drupal credit system is open source** ([contribution_records](https://www.drupal.org/project/contribution_records)) — a credit registry can start from a fork.
- **OSSDirectory is operated by BFH/CHopen with the source available** ([gitlab.com/ch-open/oss-directory](https://gitlab.com/ch-open/oss-directory)) — a catalog the project's own coalition controls, making it the natural reference implementation for the new fields and for proving multi-crawler interoperability (see [ROADMAP.md](ROADMAP.md), Phase 1).
- **Controlled forges allow platform-level integration.** There is an established relationship with [openCode.de](https://opencode.de/) (already an ally and the most advanced badge/usage-signal layer, see [ROADMAP.md](ROADMAP.md)), and a planned **Forgejo-based `opencode.ch`** (git hosting for Swiss public-sector projects) under the [Zentrum SDS](https://zentrumsds.ch/) initiative. A controlled forge can build authoring, validation, preconfigured-editor handoff, badges, and `.well-known` usage publication into the platform itself — the strongest form of "lower the effort." Because [Forgejo](https://forgejo.org/) is open source, features built for `opencode.ch` can be upstreamed and reused by every Forgejo/Codeberg instance.

## Tooling this proposal needs (new or adapted)

### Lowering the authoring barrier

The blank-page problem is the main friction. Two philosophies exist, and they are complementary:

1. **Auto-generation** (OpenCatalogi's `publiccode-action`) — a bot writes and maintains the file. Low effort, but removes maintainer judgment and re-commits stale-prone data.
2. **Assisted authoring** — pre-fill a draft, then route the maintainer through an editor with validation, hints, and AI support so they confirm/adjust. Keeps human judgment where it belongs.

This proposal favors assisted authoring. Concrete tools:

- **Preconfigured editor URL.** Export a link that opens [publiccode-editor](https://github.com/italia/publiccode-editor) pre-populated with a draft, so the maintainer does only final tweaking inside a tool that validates and hints. Two transports, modelled directly on [Metroviz](https://rstockm.github.io/Metroviz/) (a roadmap visualiser that encodes its full state in the URL):
  - _Self-contained_ — the whole draft compressed into the URL via [lz-string](https://github.com/pieroxy/lz-string)'s `compressToEncodedURIComponent` (Metroviz's `data=` param). No backend, works offline, shareable as a pure artifact. Prefer the URL **fragment** so the draft is never sent to a server. Downside: length, and the editor must support import-from-URL.
  - _External store_ — a short id/hash resolving to a stored draft, or a `source=` param pointing at a hosted JSON (Metroviz supports both). Handles large drafts and gives conversion telemetry, but reintroduces a backend dependency.
- **Crawler-assisted enrichment → open-in-editor links.** A crawler derives a _draft_ classification/supply-chain block from README, topics, and package metadata, then emits a preconfigured-editor URL for the maintainer to confirm. This closes the loop between machine-derived data and human authorship, and keeps the committed file minimal (see [PROPOSAL.md — Reducing the Maintainer Classification Burden](PROPOSAL.md#reducing-the-maintainer-classification-burden)).
- **AI-assisted classification assistant** inside the editor — reads the repo and proposes facet values to accept/adjust (already noted in [PROPOSAL.md](PROPOSAL.md) and [ROADMAP.md](ROADMAP.md)'s classification wizard).
- **Bulk onboarding, scoped by consent.** Opening MRs that add a `publiccode.yml` is welcome for a _controlled, trust-the-operator group_ (e.g. openCode.de member repos) but reads as spam on the open GitHub population. Tactics, increasing in intrusiveness: opt-in action/App (explicit consent by installation) → a preconfigured-editor _link_ dropped only where there is prior interest (FUNDING.yml, catalog listing, requested onboarding) → operator-run batch MRs reserved for trusted groups with prior announcement. See [ROADMAP.md](ROADMAP.md) for sequencing.

### Validation & types

- Extend [publiccode-parser-go](https://github.com/italia/publiccode-parser-go) and [publiccode.ts](https://github.com/publiccodeyml/publiccode.ts) to validate `supplyChain` and `classification`, and to emit **deprecation warnings** for temporal fields (`softwareVersion`, `releaseDate`, `dependsOn[].versionMin/Max`).
- A versioned **classification vocabulary artifact** (terms with `name`, `description`, `examples`, `related`, `aliases`) that the validator and editor consume, following the IPA-vocabulary model.

### Registries & discovery

- **Registry discovery tooling** around a `/.well-known/publiccode-registry.json` manifest, so any crawler can find usage registries without manual onboarding (see [EU_POLICY_CONTEXT.md](EU_POLICY_CONTEXT.md) and [ROADMAP.md](ROADMAP.md)).
- **Internal scan-and-publish tool** an organization runs to inventory deployed software and emit a `.well-known/publiccode-usage.json` (or bulk-import into a registry) — usage data lives with the deploying org, not in the project's file.
- **Funding declarations** — adopt [funding.json](https://fundingjson.org/) wholesale for `.well-known/publiccode-funding.json` (it already has a spec, validator, parser library, and crawler) and add an optional `fundingUrl` pointer; aggregate for funding-gap analysis.
- **Assessment Registry stack** — assessor publishing tooling, a reference read API (`/assessments/{url}`, `/assessors`, `/frameworks`), a catalog-side aggregator with trust-weighting and dispute display, and a shared status/framework vocabulary. Per the project's design principle, the base standard defines no metric of its own — these are third-party assessors publishing their own findings, which catalogs aggregate.
- **API conformance suites** for the Credit, Usage, and Assessment read APIs — generic OpenAPI contract tools ([Schemathesis](https://github.com/schemathesis/schemathesis), [Dredd](https://github.com/apiaryio/dredd)) run against each registry's OpenAPI spec, so this is adoption, not new code.

### Trust & identity

- **Trust-model machinery** for the `verified-domain` / `signed-attestation` / `self-reported` models: DNS/institutional **domain verification**, **sector verification** against a public-sector domain list, **attestation signing/verification** (adopt [Sigstore](https://www.sigstore.dev/) + [in-toto](https://github.com/in-toto/attestation)/[SLSA](https://slsa.dev/)), and **trust pass-through** so catalogs can filter by trust model ("government-verified only", "exclude self-reported").
- **Entity-identifier resolver** — resolve `SCHEME:VALUE` organization identifiers and do cross-registry entity matching over authoritative registries ([GLEIF LEI](https://www.gleif.org/en/lei-data/gleif-api), [ROR](https://ror.org/), Wikidata). A thin resolver, not a new registry.

### Credit

- **Turnkey credit system** — package [Drupal's `contribution_records`](https://www.drupal.org/project/contribution_records) (GPL, open source) or a derivative as a self-hostable/SaaS credit system with out-of-the-box Credit Registry API conformance, so projects can adopt vendor-credit tracking without building it. Because the Drupal code is OSS, this is an **adaptation, not a clean-sheet build**; [ecosyste-ms/funds](https://github.com/ecosyste-ms/funds) is an adjacent OSS funding/credit data source (see [ROADMAP.md](ROADMAP.md)).

## Existing building blocks for the gaps

Web research (June 2026) shows most of the new requirements can lean on mature open-source tools rather than starting from scratch. Verdicts: **Adopt** = usable directly; **Adapt** = fork/extend OSS; **Glue** = wire existing tools together (the proposal-specific logic is the connective layer); **Greenfield** = genuinely new.

**Takeaway:** the genuinely greenfield work is narrow — the three registry read APIs and their discovery, trust/sector verification, the per-deployment→organization aggregation, and the machine-readable accessibility profile. Everything else **adopts or glues** existing OSS. Two alignments are especially strong: the proposal's `supplyChain` references already match the OpenSSF stack (Security Insights, Scorecard, OSPS Baseline, Sigstore), and funding declarations can reuse funding.json wholesale.
